Alpha Release

Next week you will release your project for alpha testing. Today we will discuss all that is involved in an alpha release

What is an Alpha Release?

Purpose of an alpha release

  • Get early feedback on usability from potential end-users and other stake holders
  • Get bug reports from potential end users as opposed to just testers
    In this context, we mean that we won't be using just software or game dev type people for testing. We'll be releasing the product to average people who would be likely to use the product
  • Motivation to get your project to a usable state

Characteristics of an alpha release

  • A minimal set of features implemented
    Not all the features will be done, but a minimal set of the most important features will be at least partially working. These features should be working well enough that users can actually use the application or game in some basic way.
  • Still has bugs
    There will be both known and unknown bugs. Give the testers a list of the known bugs.
  • You should start using some form of bug tracking at this point
    You can enter bugs in your agile project management system and identify them as "issues" or "bugs" to distinguish them from user stories.

Real life story

At Intel Architecture Labs:
  • Contrast the testing done by the QA department to alpha testing done by "real world" users.
  • E-mail auto-sorting feature that passed QA but failed alpha testing.

Team Assignment

Each team member will make at least one post to the forum. Be sure to address each of the points below:
  • A UX survey. (updated from the UX testing assignment)
    Just the survey not the results. You will present the results in your report next week.
  • A test procedure. (updated from the functional testing assignment)
    This will be an extension of the one you submitted for the Functional Testing assignment. Again, this is just the test procedure, not the results.
  • A list of alpha evaluators (aka alpha testers).
    This will be an update to the list you submitted for the Functional Testing assignment. Try to recruit as many non-programmer alpha evaporators as you can.
  • A list of features ready for testing

Each team member will be graded on their post

Revised instructions for 2019

Each team member will make at least one post to the forum. Each of the documents below should be posted.
  • A UX survey. (updated from the UX testing assignment)
    Just the survey not the results. You will present the results in your report next week.
  • A test procedure. (updated from the functional testing assignment)
    This procedure should cover all the features that are now working in the alpha version of your project. Again, this is just the test procedure, not the results.
  • A list of alpha testers
    This will be an update to the list you submitted for the Functional Testing assignment. You should have at least four testers who are not members of your team, clients, or sponsors. At least two should be non-programmers / non-game designers.
  • An updated version of your design document. This update will reflect changes you have made to the design of your project.
Each team member will be graded on their post
